# RAG 개선안 문서

## 개요

현재 시스템의 검색/답변 품질을 개선하고 사용자 경험을 향상하기 위한 2단계 개선안입니다.

---

## 1. 대화이력기반질문정제 (필수) ✅ 구현 완료

### 목적
- 후속 질문(예: "그럼 안돼?", "어디에 저장?")의 맥락 손실 방지
- 벡터 검색 품질 향상
- 임베딩 모델 부하 최소화 (한 번만 실행)

### 구현 방식

**흐름:**
```
사용자 질문 + 이전 3턴 대화 이력
    ↓ LLM (gpt-4o-mini)이 정제된 단일 질문 생성
    ↓ 정제된 질문을 임베딩
    ↓ 벡터 검색 수행 (기존 방식 유지)
```

**조건:**
- 대화 이력 **있음**: LLM이 질문 정제 (1회 호출)
- 대화 이력 **없음**: 정규화만 진행 (LLM 호출 안 함)

**예시:**

```
[상황 1: 명확한 후속 질문]
사용자: "인증서가 뭐야?"
봇: "인증서는 계정을 인증하는 디지털 증명서입니다."
사용자: "어디에 저장해야 되는데"

[LLM 정제]
입력: 현재 질문("어디에 저장해야 되는데") + 이전 대화 2턴
출력: "인증서는 어디에 저장해야 하나요?"

[벡터 검색 결과]
"인증서 저장 위치 가이드" ← 정제된 질문으로 정확히 매칭
```

```
[상황 2: 첫 질문 (대화 이력 없음)]
사용자: "급여지급 중에 인증서 오류가 난다"

[LLM 정제 스킵]
정규화만 수행 → 바로 임베딩
(첫 질문은 이미 충분히 명확)
```

#### 결론
- 질문정제는 **모호하거나 축약된 후속 질문** (FU-02, FU-05, FU-06, FU-07 등)에서 유사도를 크게 향상시킴
- 프롬프트 v2 기준 **평균 유사도 +0.080 향상, 정확도 100% 유지**
- 응답시간 +560ms는 LLM 정제 호출 비용 (대화 이력 있을 때만 발생)

## 2. 역질문 + 재검색플래그 (선택) 🎯 향후 계획

### 목적
- 모호한 상황에서 명확한 역질문으로 UX 개선
- 불필요한 재검색 제거 (비용 절감)
- 정확도 향상

### 구현 방식

**흐름:**
```
사용자 질문
    ↓ LLM이 두 가지 반환:
    ├─ 정제된 단일 질문
    └─ 재검색 필요 여부 (True/False) + 역질문 필요 여부
    
    ├─ [재검색=True] → 벡터 검색 수행
    │   → 모호한 경우(다중 문맥): 역질문 생성
    │      예: "급여지급 중인가요? 물품대 지급 중인가요?"
    │   → 사용자 응답 후 기존 KB만으로 최종 답변
    │
    └─ [재검색=False] → 벡터 검색 스킵
        → 이전 검색 KB + 대화 이력으로 답변
```

**예시:**

```
[상황: 모호한 질문]
사용자: "인증서 오류가 난다"

[분석]
검색된 문서들: 급여지급, 물품대 지급, 마이페이지 등 여러 상황 가능
결과: 재검색 필요 (True) + 역질문 필요 (True)

[역질문]
봇: "급여지급 중인가요? 물품대 지급 중인가요? 
     (선택: 1번, 2번 또는 다른 상황)"

[사용자 응답]
사용자: "급여지급"

[최종 답변]
LLM에게 넘김:
- 이전 대화 3턴
- 역질문 응답 ("급여지급")
- 기존 검색된 KB (새로운 검색 안 함)
- 재검색 필요: False

봇: "급여지급 시에는 인증서를 [정해진 위치]에 등록해야 합니다..."
```

```
[상황: 명확한 후속 질문]
사용자: "인증서 저장 위치가 뭐야"
→ 재검색 필요 (True) + 역질문 불필요 (False)
→ 바로 벡터 검색 후 답변
```

### 주요 변경사항

**LLM 응답 형식:**
```json
{
  "refinedQuestion": "인증서 오류를 해결하려면 어떻게 해야 하나요?",
  "reSearchNeeded": true,
  "clarificationNeeded": true,
  "suggestedClarifications": [
    "급여 지급 중인가요?",
    "물품대 지급 중인가요?",
    "마이페이지에서 오류인가요?"
  ]
}
```

**역질문 상태 관리:**
- 호출 횟수 제한: 최대 2회
- 세션 또는 Redis에서 상태 유지
- 타임아웃: 5분 (답변 없으면 자동 진행)

### 비용 영향 (월 10,000 채팅 기준)

가정: 30% 모호한 질문 → 역질문 처리 → 재검색 회피

| 항목 | 현재 | 개선 후 | 절감 |
|------|------|--------|------|
| **임베딩** | 1,000회 | 700회 | -300 (-$0.006) |
| **질문정제** | 0 | 1,000회 | +1,000 (+$1.50) |
| **Rerank** | 300회 | 210회 | -90 (-$0.10) |
| **합계** | **$8.25** | **~$9.55** | **-$0.70** (큰 폭 절감 아님) |

**장기 효과:** 사용자 만족도 ↑ → 재질문 감소 → 실제 월간 비용 5~10% 절감 예상

### 구현 복잡도
- **난도:** 중간 ~ 높음
- **개발 일정:** 3~5일
- **필수 작업:**
  - KB 자동 분류 (급여/물품/마이페이지 등)
  - 역질문 프롬프트 설계
  - 상태 관리 로직 (세션/Redis)
  - 클라이언트 역질문 UI

### 장점
- ✅ 모호한 상황에서 정확도 크게 향상
- ✅ 불필요한 재검색 제거
- ✅ 사용자 경험 개선 (명확한 선택지 제공)
- ✅ 중장기 비용 절감 (재질문 감소)

### 단점
- ❌ 개발 복잡도 높음
- ❌ 상태 관리 필요
- ❌ 클라이언트 UI 작업 필요
- ❌ KB 분류 작업 선행 필요

### 구현 계획
- **Phase 1:** 기존 KB 분류 (3~4일)
- **Phase 2:** 역질문 로직 개발 (2~3일)
- **Phase 3:** 테스트 & 배포 (1~2일)
- **총 예상:** 6~9일

### 구현 상태
- **상태:** 🎯 향후 계획 (우선순위: 중간)
- **조건:** Phase 1 (질문정제) 안정화 후 진행 권장
- **의존성:** KB 분류 완료 필수
