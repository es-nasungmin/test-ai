# RAG 운영 발표 요약 (2026-05-11)

이 문서는 발표용 1차 요약 자료입니다.
핵심 순서는 KB 등록/운영 -> 사용자 질문 -> 챗봇 답변 -> 운영/비용/리스크입니다.

---

## 1. 한 줄 요약

- 현재 구조는 KB 운영 파이프라인과 질의응답 파이프라인이 분리되어 있고, 대량 KB 등록 시 벡터 동기화를 백그라운드 순차 처리로 전환해 운영 안정성을 개선했습니다.

---

## 2. 서비스 구조 (현재 운영 기준)

| 레이어 | 구성 | 역할 |
|---|---|---|
| Frontend | Vue 3 + Vite | 관리자 KB 운영/모니터링, 사용자 채팅 UI |
| Backend | ASP.NET Core Web API (.NET 10) | 인증, RAG 파이프라인, 운영 API |
| RDB | SQLite/MSSQL | KB, 대화, 운영 로그, 동기화 상태 저장 |
| Vector DB | Qdrant | KB 본문/예상질문 임베딩 저장 및 검색 |
| LLM | OpenAI | 질문 임베딩, 조건부 정제/rerank, 최종 답변 생성 |

---

## 3. 플로우 A: KB 등록/운영 (발표 시작점)

```text
관리자 KB 등록/수정
  -> RDB 저장 (Title/Content/Keywords/ExpectedQuestions/Visibility/Platform)
  -> 벡터 동기화 상태 관리 (pending/synced/failed)

단건 등록/수정
  -> 즉시 Upsert 동기화 시도

대량 등록(CSV)
  -> 우선 RDB 저장 + status=pending
  -> 백그라운드 큐가 pending KB를 순차 동기화
```

운영상 중요한 점

1. HTTP 요청 안에서 대량 임베딩을 다 처리하지 않으므로 타임아웃 위험을 줄였습니다.
2. 동기화 실패 건은 failed 상태로 남아 추적과 재시도가 가능합니다.
3. 관리자 화면에서 동기화 상태 칩 클릭으로 KB 단건 재동기화가 가능합니다.

---

## 4. 플로우 B: 사용자 질문 -> 챗봇 답변

```text
사용자 질문 입력
  -> 질문 정규화
  -> (필요 시) 대화 이력 기반 질문 정제
  -> 질문 임베딩 생성
  -> Qdrant 검색 (document + expected 분리)
  -> KB 후보 병합
  -> semantic + keyword 보정으로 정렬
  -> 조건부 rerank
  -> semantic threshold로 최종 통과 판정
  -> 상위 근거 컨텍스트로 GPT 답변 생성
  -> 대화/진단/저유사도 로그 저장
```

판정 정책 핵심

1. keyword는 순서 보정용
2. 최종 통과/실패는 semantic 기준
3. 저유사도 질문은 fallback 후 별도 로그 축적

## 5. 코드리뷰 관점에서 발표 시 강조할 체크포인트

### A. 정합성

1. KB 상태와 Qdrant 상태가 일치하는가
2. 동기화 실패 시 상태 전이가 명확한가
3. 수동 재동기화와 백그라운드 동기화의 충돌 가능성이 관리되는가

### B. 장애 복구

1. 서버 재기동 시 pending 재큐잉이 동작하는가
2. failed 건 재처리 경로가 운영자에게 명확한가
3. 전체 재임베딩과 단건 동기화의 책임 범위가 구분되는가

### C. 성능

1. 대량 등록 시 API 응답시간이 동기화 처리량과 분리되는가
2. 질문 처리에서 임베딩/rerank/답변 생성 구간의 지연이 분리 관측되는가
3. 동시 요청 증가 시 병목 구간(OpenAI, Qdrant, DB)이 식별되는가

### D. 보안/운영

1. admin/user 권한 경계가 API 단위로 안전한가
2. 운영 환경에서 키/시크릿 관리가 분리되었는가
3. 감사용 로그(질문/매칭/저유사도)가 운영 개선에 충분한가

## 7. 현재 리스크와 대응 방향

남은 리스크

1. 의미가 비슷한 도메인 문장 간 오탐 가능성
2. 동시 요청 시 tail latency 증가
3. 재동기화 정책(업서트 중심 vs 삭제 후 재생성)의 운영 표준화 필요

권장 대응

1. retrieval diagnostics 기반 오탐 질문군 주기적 정리
2. rerank 실행 조건/컨텍스트 토큰량 튜닝
3. 동기화 정책을 운영 규칙으로 명시
