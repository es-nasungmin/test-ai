# RAG 운영/발표 요약

이 문서는 2026-05-11 발표 기준으로 현재 RAG 구조, 최근 조정 사항, 재검증 결과, 운영 반영 판단을 한 번에 설명하기 위한 요약 자료입니다.

---

## 1. 한 줄 결론

- 품질과 일관성은 운영 반영 가능한 수준입니다.
- 다만 현재 성능은 대규모 트래픽 운영 기준으로는 아직 보수적으로 봐야 합니다.
- 특히 negative 오탐 2건은 아직 남아 있으므로, 전면 확대보다는 제한된 운영 반영이 안전합니다.

발표용 짧은 문장

"키워드는 정렬용으로만 쓰고, 최종 답변 가능 여부는 semanticScore로만 판단하도록 단순화했습니다. 그 결과 품질 일관성은 유지됐고, 구조 설명도 훨씬 명확해졌습니다."

---

## 2. 이번에 바꾼 핵심 정책

이전에는 keyword boost가 후보 정렬뿐 아니라 threshold 해석에 영향을 줄 여지가 있었습니다.

이번 변경 후에는 아래처럼 역할을 분리했습니다.

| 항목 | 역할 |
|---|---|
| semanticScore | 최종 통과/실패 판정 |
| keywordBoost | 후보 순서 보정 |
| adjustedSimilarity | 정렬 및 diagnostics 설명 |

핵심 의미

1. 키워드가 조금 겹친다고 low-quality 후보가 threshold를 억지로 넘지 못합니다.
2. 따라서 negative 질문에서 설명하기 어려운 오탐을 줄이기 쉬워집니다.
3. RetrievalDiagnostics에는 semantic, keyword, adjusted 점수가 모두 남아 운영 추적은 유지됩니다.

발표용 문장

"키워드는 순서를 정리하는 약한 신호이고, 실제 채택 여부는 semanticScore가 결정합니다."

## 3. 현재 최종 구동 구조

현재 요청 1건이 처리되는 흐름은 아래와 같습니다.

```text
사용자 질문
  -> 질문 정규화
  -> 조건부 질문 정제
  -> 임베딩 생성
  -> Qdrant 검색(document / expected 분리)
  -> KB 단위 병합
  -> keyword boost 계산
  -> adjustedSimilarity 기준 정렬
  -> 조건부 LLM rerank
  -> semanticScore 기준 threshold 판정
  -> selectedResults 상위 3개 확정
  -> GPT 답변 생성 또는 low-similarity 즉시 반환
  -> 세션/로그/통계 저장
```

구성 요소별 역할

| 단계 | 목적 |
|---|---|
| 질문 정규화 | 표현 흔들림 축소 |
| 질문 정제 | 후속 질문을 단일 의미 질문으로 복원 |
| Qdrant 검색 | 본문과 예상질문 기반 후보 수집 |
| KB 병합 | document hit와 expected hit를 같은 KB로 합침 |
| keyword boost | 비슷한 semantic 후보 사이 순서 보정 |
| rerank | 애매한 상위 후보 재정렬 |
| semantic threshold | 답변 가능 여부 최종 결정 |
| selectedResults | 실제 답변 컨텍스트로 투입될 KB 선정 |

장점

1. 구조 설명이 명확합니다.
2. diagnostics가 충분히 남습니다.
3. document/expected 분리 검색 덕분에 검색 recall이 좋습니다.
4. 질문 정제가 후속 질문 품질을 안정적으로 끌어올립니다.

주의점

1. 임베딩, 벡터 검색, rerank, 최종 답변 생성이 모두 직렬에 가까워 지연이 누적됩니다.
2. 상위 후보가 애매할 때 rerank와 answer generation이 성능 병목이 됩니다.
3. negative 오탐은 threshold 정책만으로 완전히 해소되지 않았습니다.

## 4. 재검증 결과

이번 재검증은 저장 위치를 repo 밖(/tmp)으로 분리해 실행했습니다. 즉, 테스트 결과 파일이 GitHub 커밋 후보를 오염시키지 않도록 처리했습니다.

### 4.1 100문항 라이브 벤치 재실행

실행 조건

- 소스: live KB 31건
- 문항 수: 100
- Positive / Negative: 93 / 7

재실행 결과

| 지표 | 이전 기준 | 재실행 |
|---|---:|---:|
| 시스템 성공률 | 100.00% | 100.00% |
| 품질 성공률 | 98.00% | 98.00% |
| Positive 품질 성공률 | 100.00% | 100.00% |
| Negative 품질 성공률 | 71.43% | 71.43% |
| 평균 응답시간 | 2386.1 ms | 2223.0 ms |
| P50 | 1944.0 ms | 1948.0 ms |
| P95 | 5036.0 ms | 3940.0 ms |

해석

1. semantic 기준 판정 변경만으로 negative 실패 2건이 즉시 해소되지는 않았습니다.
2. 대신 P95는 약 1.1초 개선됐습니다.
3. 즉 이번 변경은 품질을 유지하면서 구조를 단순화한 변경으로 보는 게 맞습니다.

여전히 남은 실패 케이스

| ID | 질문 | 잘못 매칭된 KB |
|---|---|---|
| N-2 | 연차 신청 승인 기준이 뭐야 | 관리자 승인 대기 해제 |
| N-4 | 사무실 와이파이 비밀번호 변경 절차 | 비밀번호 재설정 메일 미수신 |

의미

- 이 두 케이스는 키워드 gate 문제라기보다, semantic 공간에서 유사한 운영/인증 계열 표현이 겹치는 문제에 더 가깝습니다.

### 4.2 후속질문 벤치 재실행

재실행 결과

| 지표 | 이력 없음 | 이력 포함 |
|---|---:|---:|
| 정확도 | 100.0% (10/10) | 100.0% (10/10) |
| 평균 유사도 | 0.580 | 0.685 |
| 평균 응답시간 | 2189 ms | 3361 ms |

해석

1. 후속질문 정제는 여전히 정확도 100%를 유지했습니다.
2. 평균 유사도는 이력 포함 시 계속 상승합니다.
3. 대신 LLM 정제 비용 때문에 응답시간은 증가합니다.

즉, 현재 질문 정제 기능은 품질 측면에서는 유효하고, 비용은 latency 증가로 지불하는 구조입니다.

### 4.3 저장 질문 리플레이 재실행

실행 조건

- 세션 수: 26
- 베이스 질문 수: 26
- 반복 라운드: 5
- 총 케이스: 130
- 동시성: 8

재실행 결과

| 지표 | 이전 기준 | 재실행 |
|---|---:|---:|
| 순차 시스템 성공률 | 100.00% | 100.00% |
| 순차 품질 일관성률 | 100.00% | 100.00% |
| 순차 평균 지연 | 2326.3 ms | 2212.9 ms |
| 순차 P95 | 3997.0 ms | 3688.0 ms |
| 동시 평균 지연 | 1921.5 ms | 2394.9 ms |
| 동시 P95 | 3241.0 ms | 3537.0 ms |
| Throughput | 3.96 req/s | 3.27 req/s |

해석

1. 저장 질문 재실행 시 판단 일관성은 매우 안정적입니다.
2. 순차 지연은 소폭 개선됐습니다.
3. 반면 동시 부하 처리량은 오히려 낮아졌습니다.
4. 따라서 현재 병목은 검색 품질보다 동시 처리 성능에 가깝습니다.

## 5. 운영 반영 가능 여부 평가

### 5.1 품질 관점

판단: 반영 가능

근거

1. 100문항 품질 성공률 98%
2. Positive 100%
3. 저장 질문 리플레이 일관성 100%
4. 후속질문 정제 정확도 100%

즉, 사용자가 실제로 묻는 질문에 대해 "대체로 맞게 답하는가" 관점에서는 운영 반영이 가능한 수준입니다.

### 5.2 안정성 관점

판단: 반영 가능

근거

1. 100문항 시스템 성공률 100%
2. 리플레이 순차/동시 모두 시스템 성공률 100%
3. 저유사도 fallback 경로가 안정적으로 동작함

즉, 시스템이 죽거나 빈 응답을 주는 문제는 현재 크지 않습니다.

### 5.3 성능 관점

판단: 조건부 반영

현재 관찰값

1. 100문항 P95 = 3940 ms
2. 리플레이 순차 P95 = 3688 ms
3. 동시성 8 처리량 = 3.27 req/s

보수적으로 보면

- 사내 제한 사용자 수, 중간 이하 트래픽 운영에는 가능
- 전면 확대 운영이나 순간 동시 요청이 큰 환경에는 아직 보수적 접근이 필요

이유

1. P95가 여전히 3초를 넘습니다.
2. req/s 3.27은 운영 버퍼가 넉넉하지 않습니다.
3. rerank + 최종 GPT 답변 생성이 겹치는 요청에서 latency tail이 길어집니다.

### 5.4 최종 판단

| 항목 | 판단 |
|---|---|
| 품질 | 운영 가능 |
| 안정성 | 운영 가능 |
| 동시 처리 성능 | 보수적 운영 권장 |
| 전면 확대 배포 | 아직 이르다 |

최종 권고

"현재 구조는 품질과 안정성 면에서는 운영 반영 가능하지만, 성능 여유가 충분하다고 보기는 어렵습니다. 따라서 제한된 운영 반영 후 성능 튜닝을 병행하는 전략이 적절합니다."

## 6. 병목 후보 분석

현재 구조에서 비용이 큰 구간은 아래 순서로 보는 것이 합리적입니다.

1. OpenAI 임베딩 호출
2. Qdrant raw search
3. 조건부 LLM rerank
4. 최종 GPT 답변 생성

특히 tail latency를 키우는 원인 후보

1. rerank가 필요한 애매한 질문
2. answer context가 길어지는 질문
3. follow-up 정제가 추가되는 요청
4. 동시 요청 시 OpenAI 응답 편차

현재 구조적 특징

1. 품질을 위해 후보를 충분히 넓게 모읍니다.
2. 애매하면 rerank를 넣습니다.
3. 답변은 selectedResults 상위 3개를 바탕으로 생성합니다.

즉, 현재 구조는 "정확도 우선형"에 가깝고, 성능은 그 대가를 치르는 구조입니다.

## 7. 운영 전 바로 할 일

우선순위 1

1. negative 실패 2건에 맞는 expected/question 표현 확장
2. 인증/승인/비밀번호 계열 의미 충돌 KB를 추가 분리 또는 표현 보강

우선순위 2

1. rerank 스킵 조건 추가 점검
2. MergeTopK, ReRankTopK, FinalTopK 축소 실험
3. 답변 컨텍스트 길이 절감

우선순위 3

1. 동시성 8 이상 부하 테스트 반복
2. 단계별 시간 분해 로그 추가
3. OpenAI 호출 시간과 Qdrant 시간을 분리 관찰
