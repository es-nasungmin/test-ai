# 운영용 성공률 지표 정의서

이 문서는 운영에서 "성공"을 어떻게 판단할지 기준을 고정하기 위한 정의서입니다.

## 1. 지표 범위

1. 시스템 성공률
- 목적: 서비스가 오류 없이 응답을 반환하는지 측정
- 성공 조건:
  - HTTP 2xx
  - answer 필드가 비어 있지 않음
- 계산식:
  - 시스템 성공률 = 시스템 성공 건수 / 전체 요청 건수

2. 품질 성공률
- 목적: 반환된 답변이 의도에 맞는지 측정
- 성공 조건:
  - Positive 문항: isLowSimilarity=false 이고 기대 KB(또는 기대 카테고리)와 매칭
  - Negative 문항: isLowSimilarity=true
- 계산식:
  - 품질 성공률 = 품질 성공 건수 / 전체 문항 수

3. Positive 품질 성공률
- 성공 조건: Positive 문항에서 기대 KB 매칭 성공

4. Negative 품질 성공률
- 성공 조건: Negative 문항에서 저유사도 차단 성공

5. 응답 지연 지표
- 평균(ms), P50(ms), P95(ms)
- 산출 대상: 시스템 성공 요청 기준

## 2. 권장 목표치 (1차 운영)

1. 시스템 성공률: 99.5% 이상
2. 전체 품질 성공률: 90.0% 이상
3. Positive 품질 성공률: 92.0% 이상
4. Negative 품질 성공률: 80.0% 이상
5. 지연 시간: P95 3000ms 이하

## 3. 품질 판정 기준 세부

1. Positive 판정
- topMatchedKbTitle이 기대 KB 제목과 동일/유사(공백/대소문자 무시)하면 성공
- 추가 검증이 필요하면 라벨러 수동 검토로 확정

2. Negative 판정
- 업무 범위를 벗어난 질문에 대해 isLowSimilarity=true를 반환하면 성공
- 범위를 벗어난데 답변을 생성하면 실패(오탐)

## 4. 운영 리포트 최소 항목

1. 총 요청 수
2. 시스템 성공률
3. 품질 성공률(전체/Positive/Negative)
4. 평균, P50, P95 응답 시간
5. 실패 케이스 Top N (질문, 매칭 KB, 실패 사유)

## 5. 주의사항

1. 품질 성공률은 데이터셋 품질에 크게 의존함
2. 배포 전/후 같은 100문항으로 비교해야 추세가 의미 있음
3. 신규 KB 대량 반영 시 벤치 재실행 필수
